# 安全策略

## 支持范围

安全修复面向当前已发布的最新桌面版本和当前生产产品站。旧版本只有在发布说明明确列出时才继续获得安全修复；用户应优先升级到最新稳定版本。

## 私下报告漏洞

请使用仓库的 [GitHub Private Vulnerability Reporting](https://github.com/Laowang3366/excel-ai-plugin-new-feature/security/advisories/new) 提交报告。不要通过公开 Issue、讨论区或 Pull Request 披露尚未修复的漏洞、真实凭据或用户数据。

报告至少应包含：

- 受影响的版本、提交或部署面；
- 可复现步骤、预期影响和必要的最小证明材料；
- 是否涉及 Electron IPC、更新签名、Office 自动化、文件路径、远程数据处理或产品站；
- 报告者希望使用的署名方式和安全联系方式。

请使用合成数据和隔离环境验证。未经明确授权，不要访问真实用户数据、执行生产拒绝服务、修改生产文件或触发真实 Office 用户会话。

## 响应目标

维护者采用以下内部响应目标；复杂事件可能需要调整，但会在私密报告中持续同步状态：

| 阶段 | 目标 |
| --- | --- |
| 确认收到 | 2 个工作日内 |
| 完成初步分级 | 5 个工作日内 |
| Critical 临时缓解或停止发布决定 | 7 个自然日内 |
| High 修复目标 | 30 个自然日内 |
| Medium 修复或书面风险接受目标 | 90 个自然日内 |

维护者会验证复现、评估影响和受影响版本、决定是否暂停发布，并在修复可用前协调披露时间。适用时会申请 CVE 或发布安全公告。任何可能暴露的凭据、签名材料或部署令牌都应立即撤销和轮换，不能等待代码修复完成。

## 安全修复要求

- 安全修复不得在日志、测试夹具、截图或提交历史中加入真实秘密。
- 涉及敏感路径的改动必须遵守 [CONTRIBUTING.md](CONTRIBUTING.md) 的双人审查和门禁要求。
- 修复应补负向自动化测试，并更新架构、部署或数据处理文档。
- 涉及 Excel/WPS/Office 的真实验证只能在隔离测试环境执行，不得附加或关闭无关用户进程。

该渠道用于安全漏洞。普通功能问题和不包含敏感信息的缺陷可以使用公开 Issue。
