# Excel/WPS 公式解题方法论

## 目标

公式任务的目标不是寻找相似案例，也不是把大量普通函数堆叠到一个单元格中，而是根据当前工作簿事实构造“最小充分公式”：结果正确、结构精简、动态可扩展、环境兼容、能够验证并便于维护。

案例只能帮助理解局部技巧，不能作为当前任务答案。最终公式必须由当前数据结构、业务规则和验收要求推导出来。

## 一、解题步骤

### 1. 读取真实结构

先确认 Excel/WPS 连接，再读取数据源、表头、样例值、目标区域和参考答案。识别一行代表的业务粒度，以及空白、合并单元格、重复键、文本数字、日期和错误值。

不要仅凭用户描述或相似案例猜测列含义、范围大小和输出形状。

### 2. 建立任务契约

在生成公式前明确：

- 输入形状：标量、向量、记录表、宽表、层级表还是区间表。
- 输出形状：单值、一列、多列、分组结果、展开明细还是二维矩阵。
- 业务键与粒度：按什么字段匹配、筛选、分组或排序，重复记录是否重复计入。
- 约束：Excel/WPS 环境、可用函数、动态数组开关、溢出空间和目标区域。
- 验收：完整样例还是局部规律，以及空值、零、重复、无匹配和非法值的预期结果。

### 3. 抽象变换链

先写清楚“输入形状 -> 输出形状”，再选择必要阶段：

`规范化 -> 构造索引/掩码/状态 -> 筛选或映射 -> 分组/聚合 -> 展平/累积 -> 重塑 -> 输出`

不要求每个阶段都出现。删除与当前任务无关的阶段，避免为了使用高级函数而增加复杂度。

### 4. 选择最小充分方法

选择标准依次为：语义正确、环境可用、形状稳定、结构精简、可读、可验证。

- 相同源范围、条件、解析和中间计算只定义一次，使用 `LET` 复用。
- 同一规则逐项执行时，定义一次 `LAMBDA`，再交给 `MAP/BYROW/BYCOL`。
- 能由一个数组变换完成时，不拆成大量逐格公式或重复表达式。
- 简单问题使用简单公式；复杂问题才引入状态、索引、归约和重塑。

### 5. 从内到外构造

稳定构造骨架为：

`LET 定义输入 -> 规范化 -> 构造索引/掩码/状态 -> 执行局部变换 -> 展平或累积 -> 重塑 -> HSTACK/VSTACK 输出`

并非每个任务都需要所有阶段，但保留的阶段必须遵循依赖顺序。每个中间变量只承担一种职责，并能说明自己的输入和输出形状。

### 6. 写入并验证

动态数组公式只写锚点单元格。写入后从锚点回读 spill 结果，检查输出尺寸、样例、边界、错误值和当前环境兼容性。写入成功不等于结果正确。

## 二、核心方法

### 1. 先生成“索引、掩码或状态”，最后取值

复杂数组公式通常先生成控制数组，再由控制数组驱动最终取值或重塑：

- `SEQUENCE` 生成位置、日期、组内序号或连续索引。
- 广播比较生成条件掩码、重复行索引或二维关系矩阵。
- `SCAN` 生成最近有效值、累计边界、组号或组内序号。
- `XMATCH/MATCH` 把业务键、边界或全局位置转换为动态来源位置。
- `INDEX/CHOOSEROWS/FILTER/GROUPBY/PIVOTBY` 消费控制数组，完成取值、聚合或重塑。

先验证控制数组的形状、边界和业务含义，再检查最终结果。不要跳过控制数组直接硬拼最终值。

### 2. 按职责选择数组函数

| 职责 | 优先考虑 | 使用条件 |
|---|---|---|
| 规范化与去空 | `FILTER/TOCOL` | 保留有效记录并统一形状 |
| 空白层级继承、累计和组内状态 | `SCAN` | 当前结果依赖前序状态 |
| 独立逐项计算 | `MAP/BYROW/BYCOL` | 每项互不依赖 |
| 变长累积或跨区域拼接 | `REDUCE/VSTACK` | 每轮会追加行列 |
| 分组聚合 | `GROUPBY` 或 `UNIQUE + MAP/FILTER` | 按业务键产生结果 |
| 重塑 | `PIVOTBY/MAKEARRAY/TOCOL/TOROW` | 行列键决定输出形状 |
| 精确取值与定位 | `XLOOKUP` 或 `XMATCH + INDEX` | 一对一映射或需要位置 |
| 最终组装 | `HSTACK/VSTACK` | 中间数组尺寸已经对齐 |

函数越新不代表方案越好。没有前序依赖时不用 `SCAN`，输出等长时不滥用 `REDUCE`，简单查找不构造无意义的多阶段流水线。

### 3. 正则用于构造业务键、条件和索引

正则不只是展示文本的清洗工具。半结构化文本需要提取业务键、条件字段或动态索引时，若当前环境支持正则且模式稳定，应优先评估正则。

适合优先使用正则的情况：

- 字段位置或分隔符会变化，但目标格式稳定。
- 同一字段存在多个前缀、长度、分隔符或书写变体。
- 提取值还要参与 `FILTER/GROUPBY/XMATCH/INDEX/PIVOTBY`。
- 普通文本函数需要三层以上嵌套，或者同一解析逻辑会重复出现。
- 正则能减少公式长度和重复扫描，并更清楚地表达“匹配什么”。

WPS 常用 `REGEXP`，新版 Excel 使用 `REGEXEXTRACT/REGEXTEST/REGEXREPLACE`，必须按实际环境选择语法。只有固定分隔的一两步解析、正则不能减少语义层，或环境不支持正则时，才优先使用 `TEXTSPLIT/TEXTBEFORE/TEXTAFTER/FIND/SEARCH`。

提取逻辑在 `LET` 中只定义一次，生成 `parsedKey`、`conditionKey` 或 `indexKey` 后，再交给筛选、匹配、分组或重塑阶段。

### 4. 追求“最小充分公式”，不靠普通函数堆叠

精简不是追求最短字符，而是减少语义层、重复扫描和重复条件：

- 用 `LET` 消除重复子表达式。
- 用一个 `LAMBDA` 替代多段近似公式。
- 用高层数组变换替代成片逐格计算。
- 用正则替代难维护的多层文本截取链。
- 环境能力不足时，清晰辅助列优于不可维护的巨型公式。

交付前删除重复表达式、合并等价条件，并检查是否可以用正则、数组函数或 `LAMBDA` 替代重复普通函数。

## 三、约束

- 公式必须从真实数据范围和字段动态推导，不能硬编码样例结果、固定输出顺序或常量索引序列。
- 数据增删或值变化后，应只需更改数据源选区或表引用即可重算。
- 公式必须以 `=` 开头，使用英文函数名、英文逗号、半角括号、ASCII 运算符和英文双引号。
- Excel 与 WPS 的正则、`GROUPBY/PIVOTBY`、`MAP/LAMBDA` 等能力不能互相假定；回读出现 `#NAME?` 时切换兼容路线。
- 动态数组只写锚点，不把同一数组公式填满预计溢出区域。
- 不用全局 `IFERROR` 掩盖尺寸、类型、索引和引用错误；只在业务明确允许时处理无匹配。
- 不使用无边界整列动态计算，不让合并单元格直接参与数组运算。
- 单元格公式接近环境长度限制时，先消除重复、简化变换或使用清晰辅助区。

## 四、验收要求

### 1. 结果正确

- 完整样例逐项一致；局部样例符合相同规律。
- 至少手工核对一组筛选、查找或聚合结果。
- 空值、零、重复键、无匹配、首组和末组符合业务要求。

### 2. 形状正确

- 输出行列数、表头、排序和溢出方向符合预期。
- `HSTACK/VSTACK` 的数组尺寸已对齐，没有补位错误。
- 动态数组无 `#SPILL!/#VALUE!/#N/A/#REF!` 等未处理错误。

### 3. 环境可执行

- 公式在当前 Excel/WPS 中实际计算，而不只是语法看起来正确。
- 当前环境不支持的函数必须切换兼容方案，并重新回读验证。

### 4. 结构可维护

- 源范围、条件和解析没有无意义重复。
- 每个 `LET` 变量职责明确，变换顺序与业务逻辑一致。
- 没有为了得到结果而堆叠大量普通函数或复制近似表达式。
- 结果正确、结构精简、无重复计算、动态可扩展和可维护，缺一不可。

## 五、失败修正顺序

1. 先按错误类型检查溢出阻塞、数组尺寸、数据类型、匹配键和索引边界。
2. 再检查输入粒度、输出粒度、分组键和边界口径是否正确。
3. 公式能算但结构冗长时，提取重复表达式，重新评估正则、控制数组和高层数组函数。
4. 只修改出错阶段并重新验证，不无分析地轮换案例公式或整体推倒重写。

生成公式前，内部只需形成六项摘要：输入形状、输出形状、业务键、必要变换、兼容路线、验收边界。然后按最小充分结构完成公式，不为展示推理过程而增加额外步骤。
